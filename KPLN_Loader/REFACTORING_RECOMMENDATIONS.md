# KPLN_Loader — предложения по рефакторингу

## 1) Ключевые точки дублирования

### 1.1 Повторяемая логика загрузки модулей (обычный режим и ExtraNet)
В `Application.cs` есть практически одинаковые блоки:
- обход директорий/файлов;
- проверка "это dll модуля";
- `Assembly.LoadFrom`;
- ветка `IsLibraryModule` / `IExternalModule`;
- формирование похожих сообщений в лог и UI.

**Что сделать:** вынести в отдельный сервис, например `ModuleLoaderService`:
- `LoadLibraryModule(...)`
- `LoadExternalModule(...)`
- `TryLoadModuleFromDirectory(...)`
- `PublishStatus(...)`

Это уменьшит размер `Application.OnStartup`, повысит тестируемость и снимет риск расхождения поведения между режимами.

### 1.2 Повторы текста ошибок/статусов
В `Application.cs` много близких `MessageBox.Show(...)` и шаблонных логов для разных ошибок загрузки.

**Что сделать:**
- централизовать сообщения в `ErrorMessages`/`StatusMessages` (константы/ресурсы);
- сделать один helper для показа пользовательских ошибок (`ShowUserError(...)`), чтобы не дублировать заголовки, кнопки, иконки.

### 1.3 Повтор retry-логики в БД-методах
В `DBServiceAbstr.cs` очень похожий retry-паттерн в `ExecuteNonQuery` и `ExecuteQuery`.

**Что сделать:**
- общий `ExecuteWithRetry<TResult>(Func<TConnection, TResult> action, ...)`;
- единая точка политики повторов (число попыток, backoff, логирование, тексты ошибок).

## 2) Best practices и архитектурные улучшения

### 2.1 Generic-тип исключения не используется по назначению
Класс `DBServiceAbstr<TConnection, TDbException>` параметризован исключением, но в коде ловятся конкретные `MySqlException` и `SQLiteException`.

**Рекомендация:**
- ловить `TDbException` и проверять `IsRetryable(ex)` (который уже объявлен как абстрактный);
- убрать прямую зависимость абстрактного класса от конкретных провайдеров.

### 2.2 SQL-конкатенация строк
Есть запросы вида:
```csharp
$"... WHERE SystemName='{currentUser.SystemName}'"
```

**Риски:** SQL injection/ошибки экранирования, даже в desktop-сценарии.

**Рекомендация:** везде перейти на параметризованные запросы Dapper:
```csharp
WHERE SystemName = @SystemName
```

### 2.3 Синхронное ожидание Task (`.Result`)
В авторизации есть `Task.Run(...).Result`.

**Рекомендация:**
- если возможно, перевести цепочку на async/await;
- либо явно документировать, почему блокировка потока безопасна в этом месте.

### 2.4 Случайный выбор описания
`new Random()` при каждом вызове может давать повторяемость при быстрых вызовах.

**Рекомендация:**
- использовать `Random.Shared` (или singleton Random).

### 2.5 Большой `OnStartup`
`OnStartup` выполняет много обязанностей: логирование, окружение, БД, UI, загрузка модулей, подписки событий.

**Рекомендация:** выделить orchestration-методы:
- `InitializeInfrastructure(...)`
- `InitializeUser(...)`
- `InitializeRibbon(...)`
- `LoadModules(...)`
- `SubscribeEvents(...)`

## 3) Приоритетный план (быстрые победы)

1. **Вынести общий ModuleLoaderService** (максимум эффекта, меньше дублирования).
2. **Унифицировать retry в DBServiceAbstr** через общий метод.
3. **Параметризовать SQL в update/select, где ещё осталась конкатенация**.
4. **Собрать все пользовательские тексты ошибок в одно место**.
5. **Разбить `OnStartup` на небольшие методы**.

## 4) Что это даст

- Меньше дублированного кода и проще поддержка.
- Стабильнее работа БД (единая retry-политика).
- Ниже риск багов при изменениях (изменение в одном месте, а не в двух ветках загрузки).
- Понятнее код для новых разработчиков.
